# WORKFLOW_CONVENTION.md

## WORKFLOW 사용 원칙 (진입 규칙)

WORKFLOW는 스크립트나 별도 명령을 요구하지 않는다.

- `[WORKFLOW]` 선언이 유일한 진입점이다
- 선언이 없으면 일반 질의로 처리한다
- Codex는 WORKFLOW 가능성을 감지하면 사용자에게 안내만 한다
- 자동 실행이나 강제 전환은 하지 않는다

> ⚠️ 주의
>
> 이 문서는 Codex가 자동으로 인식하는 설정 파일이 아니다.
> WORKFLOW 규칙은 사용자가 Codex에게 요청 시
> 이 문서의 내용을 전제로 대화하거나,
> `[WORKFLOW]` 선언을 통해 명시적으로 활성화된다.
>
> 본 문서는 팀 내에서
> “Codex를 어떻게 사용한다”에 대한 **행동 규약**이다.

### 터미널 입력 대응 규칙 (강제)

WORKFLOW는 터미널 환경을 고려하여
사용자로부터 상세 계획 입력을 요구하지 않는다.

- 사용자는 `[WORKFLOW] 작업 요약` 한 줄만 입력한다
- 계획은 Codex가 Dry-run 단계에서 초안으로 제시한다
- 사용자 승인(Y) 없이는
  - 파일 탐색 ❌
  - 코드 분석 ❌
  - 구현 ❌

## [WORKFLOW] Prefix 규칙

사용자가 입력 메시지에 `[WORKFLOW]` prefix를 포함할 경우,
Codex는 해당 작업을 **워크플로우 기반 자동화 작업**으로 인식한다.

이 경우 Codex는 다음을 반드시 수행한다.

- 0️⃣ 작업 시작 가드 자동 실행
- Fast Track 적용 가능성 자동 판단
- 사용자 선언 없이 단계 생략 ❌
- 선언·입력 누락 시 즉시 중단

`[WORKFLOW]` prefix가 없는 요청은
일반 질의 또는 자유 응답으로 처리한다.

### 번호 규칙

- 0️⃣ ~ 9️⃣ : 워크플로우 메인 단계
- N.1, N.2 : 해당 단계의 세부 절차
- N-A, N-B : 단계 확장/보조 규칙 (분기·UX·보완)

## WORKFLOW 사전 안내 템플릿 (자동 감지 시)
```text
[WORKFLOW 사전 안내]

이 요청은 실제 코드 변경 작업으로 보입니다.

원하시면 WORKFLOW 모드로 진행할 수 있습니다.
- 설계 누락 방지
- Fast Track 가능 여부 안내
- 커밋 / PR 규칙 자동 적용

WORKFLOW로 진행할까요? (Y/N)

- Y : WORKFLOW 시작
- N : 일반 답변으로 진행
```

### WORKFLOW 사전 안내 동작 규칙

- 이 안내는 `[WORKFLOW]` 선언이 없는 경우에만 출력한다
- 사용자의 명시적 선택(Y/N) 없이는 WORKFLOW를 시작하지 않는다
- N 선택 시
  - 일반 질의 응답으로 처리
  - WORKFLOW 규칙 적용 ❌
- 이 안내는 권유 목적이며
  - 강제 전환 ❌
  - 자동 실행 ❌

### WORKFLOW 자동 감지 제한 규칙

다음 요청에는 WORKFLOW 사전 안내를 출력하지 않는다.

- 개념 설명 요청
- 코드 예제만 요구하는 질문
- 리뷰·비교·이론 설명 목적의 질의

## 0️⃣ 작업 시작 가드 (Hard Stop)

이 단계는 **모든 작업 수행 이전에 반드시 실행**된다.  
아래 조건 중 하나라도 위반되면 Codex는 **즉시 중단**한다.

### 필수 체크 항목

- `DEVELOPMENT_CONVENTION.md` 로드 여부 확인
- 현재 Git 브랜치 확인
    - 아래 브랜치에서는 **모든 Codex 작업 금지**
        - `server`
        - `main`
        - `release/*`
- `server` 브랜치 직접 push 시도 여부
    - 자동 수행 ❌
    - 사용자 **1회 명시적 확인 필수**

### 실패 시 동작

- 코드 생성, 수정, 분석 **모두 금지**
- 즉시 중단 메시지 출력

### 중단 메시지 규격

> 현재 브랜치는 보호 브랜치입니다.  
> Codex 작업을 중단합니다.

---

### Fast Track 가능성 사전 알림 규칙

Codex는 작업 시작 가드(0️⃣) 이후,  
아래 조건을 모두 충족한다고 판단되는 경우에 한해  
**Fast Track 적용 가능성을 사용자에게 알릴 수 있다.**

이 알림은 **권유나 자동 적용이 아닌, 정보 제공 목적**이다.

#### Codex가 수행할 수 있는 동작

- Fast Track 적용 가능 여부를 명시적으로 알림
- 충족된 조건을 사실 기반으로 나열

#### Codex가 해서는 안 되는 동작

- Fast Track 자동 적용 ❌
- Fast Track 강제 권유 ❌
- 사용자 선언 없이 생략 경로 진입 ❌

#### 알림 예시

```text
[Fast Track 가능성 알림]

이 작업은 코드 구조 정리 / 문서 수정처럼
영향 범위가 작은 변경으로 보입니다.

- DB 변경 없음
- 비즈니스 로직 변경 없음
- API 변경 없음

원하시면,
👉 설계·검토 과정을 건너뛰고 바로 구현 → 커밋 → PR로 진행할 수 있습니다.

단,
- PR 템플릿 작성
- 변경 내용 설명 책임
은 그대로 유지됩니다.

Fast Track으로 진행할까요? (선언 필요)
```

---

## 0️⃣-A Fast Track 규칙 (저위험 작업 분기)

Fast Track은 **저위험·비도메인 변경 작업**에 한해  
일부 설계·검증 단계를 생략하는 경로이다.

### 적용 조건 (모두 만족해야 함)

- Commit type이 다음 중 하나
    - `refactor`
    - `chore`
    - `docs`
    - `test`
- DB 스키마 변경 없음
- 도메인 상태 전이 없음
- API 시그니처 변경 없음
- 대량 데이터 생성 또는 삭제 로직 없음
- 외부 API 호출 또는 복잡한 Mocking 필요 없음

### Fast Track 적용 시 생략 단계

- 2️⃣ 엔드포인트 설계 단계
- 3️⃣ 성능 위험 판단 단계
- 4️⃣ 트랜잭션 경계 선언 단계
- 5️⃣ 상태 변경 vs 값 변경 구분 단계
- 6️⃣ 멱등성 판단 단계

### 생략 불가 단계

아래 단계는 Fast Track에서도 **반드시 수행**해야 한다.

- 0️⃣ 작업 시작 가드
- 7️⃣ 작업 수행 단계
- 8️⃣ 커밋 자동화 단계
- 9️⃣ PR 생성 전 체크리스트

### 주의

- Codex가 Fast Track을 **자동 판단해서는 안 된다**
- Fast Track 적용 여부는 **사용자 선언 기반**으로만 결정된다

### Fast Track과 PR Template 관계 규칙

Fast Track이 적용되더라도  
**PR Template 준수는 필수**이다.

Fast Track은 일부 설계·검증 단계를 생략하는 경로일 뿐,  
PR 설명 책임을 완화하거나 면제하지 않는다.


#### 적용 규칙

- Fast Track 여부와 관계없이  
  `.github/pull_request_template.md`는 **항상 사용해야 한다**
- Fast Track + 테스트 생략 조합의 경우
  - Fast Track 선언
  - 테스트 생략 사유
  - 리스크 인지 및 책임 선언
    을 **PR Template에 반드시 명시**해야 한다
- 템플릿 항목 누락 시
  - PR 생성 ❌
  - PR 병합 ❌

Fast Track PR은  
**설명이 줄어드는 PR이 아니라, 설명이 더 명확해야 하는 PR**이다.

### 🔹 Fast Track + 테스트 생략 PR 설명 강제 템플릿

- Fast Track 적용 여부: Yes
- Fast Track 적용 사유:
  - (예: refactor / docs / chore 성격의 변경)
  - 도메인 로직, DB 스키마, API 시그니처 변경 없음

- 테스트 생략 여부: Yes
- 테스트 생략 사유:
  - (예: 로직 변경 없음 / 리팩터링만 수행 / 기존 테스트 커버 범위 내)

- 리스크 인지 선언:
  - 본 PR은 Fast Track 경로로 진행되었으며,
    테스트 생략으로 인한 잠재적 리스크를 인지하고 있습니다.
  - 문제 발생 시 즉시 롤백 또는 수정 PR을 진행하겠습니다.

📌 적용 규칙 (강제)
- Fast Track + 테스트 생략이 함께 적용된 PR은
  위 템플릿 섹션이 **PR 본문에 반드시 포함**되어야 한다.
- 한 항목이라도 누락될 경우
  - PR 생성 ❌
  - PR 병합 ❌
- Fast Track PR에서 이 템플릿을 생략하는 행위는
  **규칙 위반**으로 간주한다.

## 1️⃣ 설계 단계 (Dry-run · 파일 수정 금지)

이 단계에서는 **어떠한 파일도 생성·수정해서는 안 된다**.  
Codex는 작업 수행 전에 반드시 **설계 결과를 먼저 선언**해야 한다.

### Dry-run 단계 실행 제한 (강제)

1️⃣ 설계 단계에서는 다음 행위가 금지된다.

- 디렉터리 탐색
- 파일 열기
- 코드 분석
- 테스트 실행

이 단계의 출력은
오직 **추정 계획 + 변경 범위 선언**으로 제한된다.

---

### 1.1 변경 범위 사전 요약 (필수)

Codex는 실제 코드 작업 전에 아래 항목을 **반드시 출력**해야 한다.

- 변경될 파일 목록
- 신규 생성 엔티티
- 수정 대상 엔티티
- DB 스키마 변경 여부
- API 시그니처 변경 여부

---

### 1.2 출력 형식 규칙

변경 범위 요약은 사람이 빠르게 검증할 수 있도록 **명시적으로 나열**한다.

- 엔티티 단위로 구분
- 신규 / 수정 여부 명확히 표시
- DB 변경 여부는 Yes / No 로 선언

---

### 1.3 설계 단계 규칙

- 이 단계에서는 **코드 작성 금지**
- 사용자 확인 없이 다음 단계로 진행 ❌
- 변경 범위가 불명확할 경우 **즉시 중단**

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 변경 범위를 명확히 특정할 수 없는 경우
- 엔티티 관계를 추론해야 하는 경우
- 기존 정책이나 코드 근거가 없는 경우

---

### 중단 메시지 규격

> 변경 범위를 명확히 판단할 수 없습니다.  
> Dry-run 단계에서 작업을 중단합니다.

---

## 2️⃣ 엔드포인트 설계 단계 (입력 수집 단계)

이 단계의 목적은 **API 구현에 필요한 설계 결정을 사용자로부터 명시적으로 입력받는 것**이다.  
Codex는 이 단계에서 **추측하거나 임의로 결정해서는 안 된다**.

---

## 2️⃣-A Draft → Approve 규칙 (제안 후 승인)

엔드포인트 설계 단계에서 Codex는  
단순 질문 대신 **초안(Draft)을 먼저 제시**해야 한다.

### 적용 원칙

- Codex는 분석을 통해 **권장 초안**을 제시한다
- 사용자는 초안을
    - 승인 (Y)
    - 수정
    - 거절 (N)
      중 하나로 응답한다

### 적용 대상

- 트랜잭션 필요 여부
- 멱등성 보장 방식
- DTO 필드 구성
- 페이지네이션 / 정렬 기본값

### 금지 사항

- 질문만 던지고 입력을 요구 ❌
- 초안 없이 결정 요구 ❌
- 사용자 승인 없이 자동 결정 ❌

### 예시

기존:
> 트랜잭션이 필요한가요?

변경:
> 다건 엔티티 변경이 감지되었습니다.  
> `@Transactional` 적용을 제안합니다. 승인하시겠습니까? (Y/N)

---

### 2.1 엔드포인트 후보 제시

Codex는 요구사항을 바탕으로 **엔드포인트 후보 목록을 제시**한다.

각 후보는 아래 정보를 포함해야 한다.

- HTTP Method
- Path
- API 목적 요약

규칙:
- 후보는 1개 이상 제시
- 사용자의 선택 없이 다음 단계 진행 ❌

---

### 2.2 사용자 입력 필수 항목 (API 유형별)

Codex는 선택된 엔드포인트에 대해  
아래 항목들을 **질문 형태로 사용자에게 입력받아야 한다**.

---

#### 조회 API (GET)

반드시 사용자에게 확인할 것:

- Response DTO 필드 목록
- 페이지네이션 필요 여부
- 정렬 기준
- 기본 정렬 방향

규칙:
- 엔티티 직접 반환 ❌
- 입력 없이 자동 결정 ❌

## 2️⃣-B DTO 필드 추천 규칙 (조회 API 전용)

조회 API의 Response DTO 필드는  
아래 절차에 따라 결정한다.

### 결정 절차

1. Codex는 엔티티 필드를 기반으로 DTO 초안을 생성한다
2. 민감 정보는 자동 제외한다
    - 예: password, internalId, secretKey 등
3. Codex는 DTO 초안을 사용자에게 제시한다
4. 사용자는 제외 또는 추가할 필드만 선택한다

### 규칙

- 엔티티 직접 반환 ❌
- 사용자 최종 승인 없는 DTO 확정 ❌
- 민감 정보 포함 ❌

### 목적

- 사용자 입력 부담 감소
- 보안 사고 예방
- DTO 설계 일관성 확보

---

#### 생성 API (POST)

반드시 사용자에게 확인할 것:

- 함께 생성되어야 하는 엔티티 목록
- HTTP Status (`200` 또는 `201`)
- 생성 시 수행되어야 하는 비즈니스 로직 존재 여부
- 멱등성 보장 방식

규칙:
- “단순 저장” 여부를 Codex가 판단 ❌
- 생성 이후 상태 정의는 사용자 입력 필수

---

#### 갱신 API (PUT / PATCH)

반드시 사용자에게 확인할 것:

- `PUT` 또는 `PATCH` 중 선택
- 원자적 갱신 여부
- HTTP Status (`200` 또는 `204`)
- 멱등성 보장 방식

규칙:
- 불변 필드 수정 여부는 사용자 판단 필요
- 상태 전이 여부는 이후 단계에서 재검증

---

#### 삭제 API (DELETE)

반드시 사용자에게 확인할 것:

- Soft Delete 필요 여부
- 실제 삭제 가능한 데이터인지 여부

규칙:
- Codex가 “지워도 될 것 같다” 판단 ❌
- 운영/이력 데이터 여부는 사용자 입력 필수

---

### 2.3 입력 완료 조건

아래 조건을 모두 만족해야 다음 단계로 진행할 수 있다.

- 엔드포인트 1개 이상 선택됨
- 해당 API 유형의 필수 항목이 모두 입력됨
- 모호하거나 미정 상태의 항목이 없음

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 사용자 입력이 누락된 경우
- 설계 항목이 “나중에 결정” 상태인 경우
- 도메인 정책을 Codex가 추론해야 하는 경우

---

### 중단 메시지 규격

> API 설계 입력이 완료되지 않았습니다.  
> 사용자 입력을 기다리며 작업을 중단합니다.

---

## 3️⃣ 성능 위험 판단 단계 (경고 + 확인)

이 단계의 목적은 **구현 전에 성능·운영 리스크를 사전에 노출**하는 것이다.  
Codex는 이 단계에서 **최적화를 시도하거나 구조를 변경해서는 안 된다**.  
오직 **위험 요소를 탐지하고 사용자에게 알리는 역할**만 수행한다.

---

### 3.1 성능 위험 검토 대상

Codex는 설계 단계 결과를 바탕으로 아래 항목을 점검해야 한다.

- 대량 row에 영향을 주는 조회 / 갱신 / 삭제 가능성
- 인덱스가 존재하지 않을 가능성이 있는 컬럼 사용
- 다건 UPDATE / DELETE 가능성
- 관리자 API지만 실시간 호출될 가능성
- 페이징 없는 전체 조회 가능성

---

### 3.2 위험 탐지 시 동작

위 항목 중 **하나라도 해당**하면 Codex는 다음을 수행한다.

- 성능 위험 항목을 **명시적으로 나열**
- “위험함 / 주의 필요 / 안전” 중 하나로 분류
- 사용자에게 **진행 여부 확인 요청**

Codex는 이 단계에서  
“괜찮을 것 같다”, “문제 없어 보인다”와 같은 판단을 **절대 하지 않는다**.

---

### 3.3 사용자 확인 필수 상황

아래 경우에는 사용자 확인 없이는 다음 단계로 진행할 수 없다.

- 페이징 없는 조회
- 다건 UPDATE / DELETE
- 배치/집계/통계 성격의 API

---

### 3.4 진행 조건

다음 중 하나를 만족해야 다음 단계로 진행할 수 있다.

- 성능 위험 없음으로 판단됨
- 성능 위험 있음 → 사용자 명시적 진행 승인

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 성능 위험이 있으나 사용자 확인을 받지 못한 경우
- 위험 여부를 판단하기 위한 정보가 부족한 경우
- 구조적 리스크를 Codex가 추론해야 하는 경우

---

### 중단 메시지 규격

> 성능 위험 가능성이 있습니다.  
> 사용자 확인 없이는 작업을 진행할 수 없습니다.

---

## 4️⃣ 트랜잭션 경계 선언 단계 (선언 강제)

이 단계의 목적은 **API 구현 전에 트랜잭션의 필요성과 범위를 명시적으로 확정**하는 것이다.  
Codex는 이 단계에서 **트랜잭션을 자동으로 추론하거나 임의로 설정해서는 안 된다**.

---

### 4.1 사용자 입력 필수 항목

Codex는 아래 항목을 **질문 형태로 사용자에게 입력받아야 한다**.

- 이 API는 트랜잭션이 필요한가?
- 트랜잭션 범위는 단일 엔티티인가, 다중 엔티티인가?
- 외부 연동(API 호출, 메시지 발행, 이벤트 처리)이 포함되는가?

규칙:
- 하나라도 미정 상태이면 다음 단계 진행 ❌
- “일단 넣어두자”와 같은 판단 ❌

---

### 4.2 트랜잭션 선언 규칙

사용자 입력을 바탕으로 Codex는 아래 내용을 **명시적으로 선언**해야 한다.

- 트랜잭션 사용 여부: O / X
- 트랜잭션 범위: 단일 / 다중
- 외부 연동 포함 여부: O / X

이 선언 없이 코드 작성 ❌

---

### 4.3 적용 규칙

- 트랜잭션이 필요 없는 API에 `@Transactional` 적용 ❌
- 엔티티 변경 시 트랜잭션 선언 필수
- 외부 연동이 포함되는 경우
    - 트랜잭션 경계 분리 여부를 명시해야 함
    - 이벤트/리스너 구조 사용 여부를 이후 단계에서 재검증

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 트랜잭션 필요 여부를 판단하기 위한 정보가 부족한 경우
- 외부 연동이 있는데 트랜잭션 경계를 정의하지 않은 경우
- 도메인 규칙을 Codex가 추론해야 하는 경우

---

### 중단 메시지 규격

> 트랜잭션 경계를 명확히 선언할 수 없습니다.  
> 사용자 입력이 필요하여 작업을 중단합니다.

---

## 5️⃣ 상태 변경 vs 값 변경 구분 단계 (강제)

이 단계의 목적은 **Update / Delete API가 단순 값 수정인지, 도메인 상태 전이인지 명확히 구분**하는 것이다.  
Codex는 이 구분 없이 구현을 진행해서는 안 된다.

---

### 5.1 사용자 입력 필수 항목

Codex는 아래 질문을 **명시적으로 사용자에게 입력받아야 한다**.

- 이 변경은 **단순 필드 값 수정**인가?
- 이 변경은 **도메인 상태 전이**인가?

규칙:
- 둘 중 하나를 반드시 선택
- “둘 다일 수도 있음”, “상황에 따라 다름” ❌

---

### 5.2 값 변경(Value Change) 규칙

사용자가 **단순 값 변경**으로 선택한 경우:

- setter 기반 수정 허용
- 불변 필드 수정 여부를 별도로 확인
- 변경은 **명시적 필드 단위**로 수행

주의:
- 누적 값 증가, 조건부 변경이 포함되면 값 변경 ❌

---

### 5.3 상태 변경(State Transition) 규칙

사용자가 **상태 전이**로 선택한 경우:

- setter 기반 수정 ❌
- 반드시 **의미를 가진 도메인 메서드** 사용
    - 예: `close()`, `activate()`, `expire()`
- 상태 전이 조건(사전 조건 / 사후 상태)을 코드로 명시

규칙:
- 상태 전이는 항상 **비즈니스 행위**로 취급
- 상태 전이 로직은 Service 또는 Domain에 명시적으로 존재해야 함

---

### 5.4 재검증 규칙

Codex는 아래 경우 **재확인 또는 중단**해야 한다.

- 값 변경으로 선택했으나 조건 분기 로직이 포함된 경우
- 상태 전이로 보이는데 setter 수정이 요구되는 경우
- 기존 도메인 규칙과 충돌하는 경우

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 상태 변경 / 값 변경을 명확히 구분할 수 없는 경우
- 상태 전이 규칙을 Codex가 추론해야 하는 경우
- 도메인 메서드 정의 근거가 없는 경우

---

### 중단 메시지 규격

> 변경 유형을 명확히 구분할 수 없습니다.  
> 상태 전이 여부에 대한 사용자 입력이 필요합니다.

---

## 6️⃣ 멱등성 판단 단계 (선언 강제)

이 단계의 목적은 **중복 요청·재시도 상황에서도 시스템 상태가 안전하게 유지되도록**  
멱등성 보장 방식을 **명시적으로 확정**하는 것이다.  
Codex는 이 단계를 **자동 추론으로 통과해서는 안 된다**.

---

### 6.1 멱등성 검토 대상 조건

아래 조건 중 **하나라도 해당하면 멱등성 검토 대상**이다.

- 재시도 가능 API
- 네트워크 지연 / 타임아웃 가능성 존재
- 관리자 반복 클릭 가능성 존재
- 상태 생성 / 변경을 수행하는 API

---

### 6.2 사용자 입력 필수 항목

Codex는 멱등성 검토 대상인 경우  
아래 질문에 대해 **사용자에게 명시적으로 입력받아야 한다**.

- 이 API는 중복 요청이 발생할 수 있는가?
- 중복 요청 시 **동일한 상태를 반환**해야 하는가?
- 요청 단위를 구분해야 하는가?

---

### 6.3 멱등성 보장 방식 선택 (필수)

Codex는 아래 중 **하나를 반드시 선택하여 선언**해야 한다.

#### ① Idempotency-Key 사용
- 요청 단위 구분이 필요할 경우
- 재시도 시 동일 요청을 식별해야 할 경우

#### ② 논리적 멱등성 보장
- 여러 번 호출되어도 결과 상태가 동일한 경우
- DB 제약 / 상태 체크로 중복 방지 가능할 경우

#### ③ 판단 불가 → 중단
- 중복 요청 정책을 명확히 정의할 수 없는 경우

---

### 6.4 선언 규칙

- 멱등성 선언 없이 코드 작성 ❌
- “아마 한 번만 호출될 것”이라는 가정 ❌
- 관리자 API라서 안전하다는 판단 ❌

Codex는 반드시 **선택 결과와 사유를 함께 출력**해야 한다.

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 멱등성 정책을 Codex가 추론해야 하는 경우
- 사용자 입력 없이 멱등성 방식을 결정해야 하는 경우
- 상태 생성 / 변경인데 멱등성 논의가 누락된 경우

---

### 중단 메시지 규격

> 멱등성 보장 방식을 확정할 수 없습니다.  
> 사용자 입력이 필요하여 작업을 중단합니다.

---

## 7️⃣ 작업 수행 단계 (Implementation Phase)

이 단계부터 **실제 코드 생성 및 수정이 허용**된다.  
이전 단계(0~6)의 선언·입력이 모두 완료되지 않았다면 이 단계로 진입할 수 없다.

---

### 7.1 작업 범위 제한

Codex는 아래 범위 내에서만 작업할 수 있다.

- 사전 선언된 파일 목록
- 사전 선언된 엔티티 / DTO / Repository / Service / Controller

규칙:
- 선언되지 않은 파일 생성 ❌
- 선언되지 않은 엔티티 수정 ❌
- “필요해 보여서 추가” ❌

---

### 7.2 구현 규칙

- `DEVELOPMENT_CONVENTION.md` 준수 필수
- `TEST_CONVENTION.md` 준수 필수
- 상태 전이는 반드시 **도메인 메서드**로 구현
- 트랜잭션 경계는 **선언된 범위**만 사용

---

### 7.3 테스트 작성

- 6️⃣ 단계에서 선언한 테스트 범위에 따라 테스트 작성
- 테스트 선언과 다른 테스트 작성 ❌
- 테스트 생략을 선언한 경우
  - 테스트 코드 생성 ❌
  - 사유는 PR 설명에 반드시 포함
- Fast Track 경로로 테스트 생략이 선언된 경우에도
  - PR 설명 책임은 완화되지 않는다

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 작업을 중단한다.

- 선언된 범위를 벗어난 구현이 필요한 경우
- 컨벤션과 충돌하는 코드가 필요한 경우
- 기존 도메인 규칙을 추론해야 하는 경우

---

### 중단 메시지 규격

> 선언되지 않은 구현이 필요합니다.  
> 작업 범위를 벗어나 자동 구현을 중단합니다.

---

## 7️⃣-A 파일 추가 예외 규칙 (명시적 허용 / 금지)

기본 원칙:
- 7️⃣ 작업 수행 단계에서는 **사전 선언되지 않은 파일 생성은 금지**된다.
  - “사전 선언”이란 1️⃣ 설계 단계에서 파일 생성이 명시된 경우를 의미한다

다만 아래 경우에 한해,
**사용자 명시 승인 또는 사전 선언이 있으면 예외적으로 허용**할 수 있다.

---

### 허용 가능 예외 (조건부)

아래 항목은 **사전 선언 또는 사용자 승인**이 있을 경우에만 허용된다.

- DTO 추가
  - 기존 엔드포인트 설계 단계(2️⃣)에서
    Response / Request DTO 필요성이 명시된 경우
- 테스트 보조 클래스
  - 테스트 전용 Fixture / Stub / Test Helper
  - `TEST_CONVENTION.md` 범위 내 클래스
- 설정 파일
  - application.yml / application-*.yml
  - 테스트용 config 파일

---

### 절대 금지 항목

아래 경우는 **어떠한 상황에서도 자동 생성 금지**이다.

- 선언되지 않은 도메인 엔티티 추가
- 선언되지 않은 Repository / Service / Controller 생성
- 운영 환경 설정 파일의 암묵적 변경
- “필요해 보여서” 생성되는 보조 클래스

---

### Codex 동작 규칙

- 파일 추가가 필요한 경우
  - Codex는 **즉시 중단**하고
  - 추가하려는 파일 종류와 목적을 **명시적으로 설명**해야 한다
- 사용자 승인 없이
  - 파일 생성 ❌
  - 패키지 구조 변경 ❌

---

### 중단 메시지 예시

```text
[7️⃣ 작업 수행 단계 중단]

사전 선언되지 않은 파일 생성이 필요합니다.
DTO / 테스트 보조 / 설정 파일 중 어느 유형인지 명확하지 않습니다.

사용자 승인 없이는 파일을 추가할 수 없습니다.
```

## 8️⃣ 커밋 자동화 단계

이 단계의 목적은 **작업 내용을 명확한 커밋 단위로 기록**하는 것이다.  
Codex는 커밋 및 PR 생성 시 반드시  
`docs/COMMIT_CONVENTION.md`를 **준수해야 한다**.

---

### 8.1 커밋 생성 규칙

- 커밋 단위는 **기능 단위**
- 커밋 메시지는 변경 의도를 명확히 드러내야 한다
- 커밋 메시지는 `docs/COMMIT_CONVENTION.md` 규칙을 따른다
- 보호 브랜치(`server`, `main`) 직접 커밋 ❌

---

### 8.2 커밋 메시지 규칙

- 커밋 메시지 형식은 `COMMIT_CONVENTION.md`를 따른다
- 제목 + 본문 구조 권장
- 제목은 **명령형** 사용
- 테스트 생략 시
    - 커밋 본문 또는 PR 본문에 **사유 명시 필수**

---

### 중단 조건

아래 중 하나라도 해당하면 Codex는 **커밋을 생성하지 않는다**.

- `COMMIT_CONVENTION.md` 규칙을 충족하지 못하는 경우
- 테스트 범위 선언과 커밋 내용이 불일치하는 경우
- 보호 브랜치에서 커밋 시도한 경우

---

## 9️⃣ PR 생성 전 최종 체크리스트 (Gate)

PR 생성 직전에 Codex는 **반드시 아래 체크리스트를 출력**해야 한다.

- [ ] 브랜치 안전함 (server 아님)
- [ ] 직접 push 여부 확인됨
- [ ] 영향 범위 확인 완료
- [ ] 테스트 존재 또는 생략 사유 명시
- [ ] 롤백 가능성 검토 완료

모든 항목이 충족되지 않으면 **PR 생성 금지**

---

### PR 생성 규칙

- PR 제목 및 내용은 `docs/COMMIT_CONVENTION.md`를 따른다
- PR 템플릿은 `.github/pull_request_template.md` 파일을 **반드시 사용**한다
- 템플릿 항목 누락 시 PR 생성 ❌
- 체크리스트 미충족 시 PR 생성 ❌

---

## 🔴 실패 전략 — “모른다” 규칙

자동화에서 가장 위험한 것은 **추측 구현**이다.  
Codex는 확실한 근거 없이 작업을 이어가는 것을 **실패가 아니라 오류**로 간주한다.

아래 상황에서는 Codex는 **즉시 작업을 중단**해야 한다.

### 즉시 중단 대상 상황

- 엔티티 관계가 불명확한 경우
- 기존 도메인 규칙을 추론해야 하는 경우
- 정책 문서 또는 코드 근거가 없는 경우
- 사용자 입력이 필요한 판단을 Codex가 대신 내려야 하는 경우
- “일반적으로”, “보통은”, “아마도”와 같은 가정이 필요한 경우

중단은 **정상 동작**이며,  
추가 질문 없이 자동 구현을 이어가는 것은 **명백한 규칙 위반**이다.

---

### 중단 메시지 작성 규칙 (공통)

Codex가 작업을 중단할 때 출력하는 메시지는  
아래 정보를 **반드시 포함**해야 한다.

#### 필수 포함 항목

- 중단된 단계 번호
- 중단된 단계 번호는 반드시 0️⃣ ~ 9️⃣ 또는 N-A 형식을 사용한다
- 중단 사유 요약
- 감지된 구체적 트리거 (사실 기반)

#### 금지 사항

- 추측 표현 사용 ❌  
  (예: “문제가 있을 수 있습니다”, “위험해 보입니다”)
- 모호한 원인 제시 ❌  
  (예: “판단이 어렵습니다”)
- 해결책 또는 구현 방향 제시 ❌  
  (판단 유도 금지)

Codex는 **사실만 서술**하고,  
다음 행동은 **사용자가 결정**하도록 남겨야 한다.

### 번호 표기 주의 사항

- 이모지 단계 표기(`0️⃣ ~ 9️⃣`)와 보조 규칙 표기(`N-A`)는 **결합하지 않는다**
- 이모지 + 하이픈(`-`) 조합은 일부 환경에서 렌더링이 깨질 수 있다
- 따라서 보조 규칙은 항상 ASCII 표기(`N-A`)를 사용한다

---

### 중단 메시지 출력 형식 (표준)

```text
[{단계 번호} 중단]

{감지된 구체적 트리거 설명}
{중단 사유 요약}

사용자 입력 없이는 작업을 진행할 수 없습니다.
```

중단 메시지 예시
```text
[3️⃣ 성능 위험 판단 단계 중단]

페이징 없는 전체 조회(findAll) 가능성이 감지되었습니다.
대량 데이터 조회로 인한 성능 위험이 있어
사용자 확인 없이는 작업을 진행할 수 없습니다.
```
```text
[5️⃣ 상태 변경 판단 단계 중단]

도메인 상태 전이 여부를 판단하기 위한 규칙이 명시되어 있지 않습니다.
상태 변경인지 값 변경인지 사용자 입력이 필요합니다.
```

핵심 원칙
- 중단은 실패가 아니다
- “모른다”를 말하는 것은 가장 안전한 자동화 행동이다
- Codex는 항상 사람의 판단을 보조한다

---

## 최종 원칙 요약

- 자동화는  
  **가드 → 설계 → 검증 → 선언 → 구현 → 커밋 → PR**  
  순서로만 진행한다.
- 중단은 실패가 아니라 **정상 동작**이다.
- Codex는 항상 **사람의 판단을 보조**한다.
